# B200 Hardware Target

The QR v2 leaderboard target is NVIDIA B200. Treat every local CPU run and every
non-B200 CUDA run as a smoke test unless proven otherwise.

Working B200 assumptions:

- GPU: NVIDIA B200 SXM
- Architecture: Blackwell
- HBM: 180GB HBM3e-class memory
- Memory bandwidth: up to 8TB/s
- Tensor Cores: fifth generation
- Interconnect: fifth-generation NVLink in HGX/DGX systems

These numbers are for tuning intuition, not correctness. The QR checker is
unchanged across machines, but performance decisions should be made on B200.

## Per-GPU Peak Rates

The table below uses NVIDIA HGX B200 8-GPU system specifications converted to
per-GPU values. NVIDIA reports some Tensor Core numbers as sparse peaks; dense
is half of the sparse value for those rows.

```text
resource                         per-GPU peak
HBM capacity                     180 GB
HBM bandwidth                    8.0 TB/s
NVLink GPU-to-GPU bandwidth      1.8 TB/s

FP64 / FP64 Tensor Core          37 TFLOP/s
FP32 CUDA core                   75 TFLOP/s
TF32 Tensor Core dense           1,125 TFLOP/s
TF32 Tensor Core sparse          2,250 TFLOP/s
FP16/BF16 Tensor Core dense      2,250 TFLOP/s
FP16/BF16 Tensor Core sparse     4,500 TFLOP/s
FP8/FP6 Tensor Core dense        4,500 TFLOP/s
FP8/FP6 Tensor Core sparse       9,000 TFLOP/s
FP4 Tensor Core dense            9,000 TFLOP/s
FP4 Tensor Core sparse           18,000 TFLOP/s
```

For QR v2, correctness is FP32-style compact Householder. Low precision is only
useful if it is internal and the final `(H, tau)` passes the official checker.
That means scalar FP32 and FP32-accurate update paths matter more than the huge
FP4/FP8 headline numbers.

## Roofline Ridge Points

Approximate ridge point:

```text
ridge FLOP/byte = peak FLOP/s / HBM byte/s
```

Using 8TB/s HBM bandwidth:

```text
path                         ridge FLOP/byte
FP64 / FP64 Tensor Core       4.625
FP32 CUDA core                9.375
TF32 Tensor Core dense      140.625
FP16/BF16 dense             281.250
FP8/FP6 dense               562.500
FP4 dense                 1,125.000
```

Interpretation for QR:

- Work below roughly `9.4` FP32 FLOP/byte is HBM-bandwidth bound even before
  considering synchronization and launch overhead.
- Householder panel factorization is likely latency/synchronization sensitive.
- Trailing updates need to become GEMM-like to approach compute rooflines.
- TF32 and lower precision have high ridge points, so they require very high
  arithmetic intensity to pay off and still risk failing FP32 QR checks.
- For dense easy inputs, a blocked trailing update should be measured against
  FP32 and possibly TF32 rooflines.
- For hard inputs, correctness is the first constraint; a lower roofline but
  stable FP32 path is preferable to a fast failing path.

## What To Measure With NCU

Collect these for candidate B200 kernels:

```text
achieved occupancy
SM active / SM throughput
DRAM throughput
L2 throughput and hit rate
global load/store efficiency
shared-memory throughput and bank conflicts
registers per thread
shared memory per CTA
warp stall reasons
number of launches
kernel duration by QR phase
```

Then classify the kernel:

```text
memory-bound: DRAM is high, SM utilization is low
compute-bound: SM/tensor utilization is high, DRAM is not the limiter
latency/sync-bound: neither DRAM nor compute is saturated, stalls dominate
launch-bound: many short kernels, especially small n
```

## Consequences For QR

- Optimize for high memory bandwidth and large batches, especially
  `batch=640, n=512`.
- Keep trailing updates GEMM-like where possible.
- Avoid many tiny launches; launch overhead matters on small shapes.
- Watch shared memory and register pressure when increasing panel width.
- For FP32-dominant work, the practical ridge point is around 9.4 FLOP/byte.
- Treat TF32/FP16/BF16/FP8/FP4 paths as experimental until residuals pass.
- Do not assume CPU or Apple/MPS timing predicts B200 behavior.
- Do not choose a dispatch path from CPU results alone.
- Do not use CUDA streams in submission code.

## B200-First Panel Candidates

Use these as starting points for autotune:

```text
n=32:       special small-n path, candidates 1,2,4,8,16
n=176/352:  8,16,32
n=512:     16,32,64
n=1024:    16,32,64
n>=2048:   32,64,128
```

Run:

```bash
uv run --group practice python -m autotune.sweep \
  --hardware b200 \
  --n 512,1024 \
  --batch 16 \
  --cases dense,rankdef,nearrank,clustered,rowscale,nearcollinear,mixed \
  --variants python_geqrf,python_blocked,cholesky_probe \
  --block-sizes auto \
  --repeats 5 \
  --output results/b200_dispatch_sweep.csv
```

`--block-sizes auto` uses the B200 hardware profile in `autotune/hardware.py`.
